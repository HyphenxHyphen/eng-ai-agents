# 1. Deep document understanding vs naive chunking (10 pts)
Deep document understanding outperforms fixed-size chunking in speed and accuracy by using hierarchical structures that resemble classical storage and query methods. Fixed-size chunking breaks down large chunks of data with no regard to the internal structure of the data, which destroys the context of the data and thus impedes future queries. With fixed-size chunking, a chunk may slice sentences in half, cutting off important context, such as what a particular number is referring to. The result is increased hallucination/mistakes as information is returned at query. Deep document understanding instead parses the structure of input data, and stores data hierarchically with contextual tags, preventing semantic drift. Retrieval fidelity is improved by searching for relevant data indices before searching for specific information, rather than attempting to search the entire dataset all at once (flat indexing, used by fixed-size chunking). 

Speed, accuracy, and quality come at a significantly higher cost to the initial data input process. Fixed-size chunking, as the name suggests, chunks data at a fixed size. It’s fast and simple. On the other hand, layout-aware systems require higher technical complexity, using computer vision and optical character recognition to read in the first place, and it is a much heavier task than reading the raw text directly. Using CV and OCR, a system like DeepDoc can recognize layout elements such as headers, footers, tables, and graphs.

Systems like DeepDoc are trained on generic, common data structures and layouts and may struggle to parse uncommon/artistic layouts. Nested tables and multi-column layouts with unclear borders may lead to wrongly fragmented or merged data, as the system only tries its best to partition data, but does not actually check for the correctness of the data as it parses. The initial upfront cost of deep document understanding is also a bottleneck for scalability, as using CV and OCR on millions of documents presents significant compute requirements. Using OCR without double-checking the result may also generate noise or bad text on low-quality scans.

# 2. Chunking strategy: template vs semantic (10 pts)
Template-based chunking is rigid and deterministic, and is best suited for documents with a known grammar/structure. It uses predefined absolute rules to break down a document by structures like HTML tags or JSON schemas. Embedding-driven chunking attempts to keep chunks topic-specific. By calculating the cosine similarity between sentences and paragraphs, it creates chunks based on semantic boundaries. 

For highly structured documents, embedding-driven chunking fails. Embedding-driven chunking operates on the assumption that data is 1D, and may lose context when parsing 2D graphs. Additionally, the headers of a graph may be segmented into their own chunk if the system determines that the text headers and the numeric entries are linguistically dissimilar. 

For loosely structured corpora, such as chat logs where conversation topics shift rapidly, template-based chunking fails. Template-based chunking may split a conversation by user, but that would omit the replies and conversational context between multiple users. 

# 3. Hybrid retrieval architecture (10 pts)
The use of hybrid retrieval attempts to combine lexical, vector, and re-ranking methods to cover the weaknesses of each individual method. The BM25 is a probabilistic model that excels at finding specific phrases and proper nouns, but fails completely if there is no lexical overlap in the search query and the dataset. The BM25 cannot detect “relevant” text or synonyms, only exact phrases. An exact failure case would be if a user searched for “dogs” but the documents only used “canines.” Using vectors for semantic recall has the exact opposite problem. This method maps text by their semantic and contextual similarity, which is good at detecting synonyms or related text, but has difficulty locating exact phrases and codes. An exact failure case would be if a user searched for “Error 404” and the query returned a “List of Errors” document instead of the document relating specifically to Error 404.

Hybrid combines the two, but requires more than combined processing power. Additionally, if the lexical and vector components return an abundance of irrelevant documents, and the re-ranker has a small context window, the re-ranker may actually filter out the most relevant article. RAGFlow attempts to alleviate this issue using reciprocal rank fusion and re-ranking with a second search through the fused list. 

# 4. Multi-stage retrieval pipeline (10 pts)
A multi-stage pipeline is superior in recall than a single-pass due to its ability to search through millions of documents with an initial course search before refining those results with finer analysis. However, there is a significant tradeoff in latency, as single-pass ANNs using precomputed indice values are computationally much cheaper than performing multiple stages of checks to ensure recall. Additionally, multi-stage systems host the risk of error propagation. An issue at any stage will propagate downwards and cause all subsequent stages to fail too. For example if the initial ANN within a multi-stage system fails to include any relevant documents, the following re-ranker will fail too. An erroneous attempt at refining the user's query at the highest level will cause error propagation through the entire system.

# 5. Indexing strategy and storage backends (10 pts)
Design criteria for selecting a backend must prioritize retrieval latency, which is the time it takes to find information, and indexing throughput, which is the speed at which new data is added. An Elasticsearch-like hybrid store is favored for production workloads that require a combination of keyword matching and semantic search because it utilizes BM25 ranking to find exact terms alongside vector capabilities. A vector-native database is best for high-dimensional similarity tasks where the goal is to find semantic or conceptual similarity rather than specific words, though it lacks precise filtering. Graph-augmented stores are best for complex reasoning where the relationship between entities, such as how one thing is connected to another, is more important than the text itself. A significant failure mode for hybrid stores is the difficulty in mathematically balancing a keyword score against a vector score. Vector-native systems can fail when a search term is so unique that the mathematical model has no relevant neighborhood to search. Graph-augmented stores fail when there exists super-nodes that are connected to so many other nodes that the system consumes all available memory or processing time and crashes.

# 6. Query understanding and reformulation (10 pts)
Query transformation is critical because of the semantic gap between imprecise human inputs and the rigid computer data storage. Expansion adds synonyms to a search to ensure a query for "automobile" also finds results for "car," while decomposition breaks a complex multi-part question into smaller, solvable pieces. A static query to retrieve pipeline is a one-shot process that is fast and cheap but fragile, as it cannot recover if the initial search fails to find relevant context. In contrast, iterative query refinement uses an agent-driven loop to check the quality of results and try again if the information is missing. The trade-off is that iterative refinement significantly increases latency and cost due to multiple calls to a large language model. Query expansion, as discussed before, can cause error propagation. It does so by setting too many synonyms which makes the search so broad that it returns irrelevant noise, confusing the system.

# 7. Knowledge representation layer (10 pts)
Dense vector space represents information as a list of numbers that plot a point in a multi-dimensional map, which is excellent for finding similar concepts but offers poor retrieval explainability because humans cannot easily understand why a specific coordinate was chosen. A relational schema stores data in traditional tables with rows and columns, providing perfect explainability and strict logic but failing at compositional reasoning because it cannot easily link ideas that were not explicitly predefined in the table structure. Knowledge graphs use nodes to represent things and edges to represent relationships, which excels at compositional reasoning by allowing the system to hop from one fact to another across the entire database, but as entries in a knowledge graph increase, searching for a connection becomes computationally expensive, which represents a scalability issue.

# 8. Data ingestion pipeline architecture (10 pts)
A robust ingestion system for RAGFlow begins with the connection phase, where the system pulls data from heterogeneous sources like S3 buckets, local files, or emails and stores them in a decoupled staging area to separate raw ingestion from heavy processing. The parsing stage follows, utilizing specialized models such as DeepDoc to perform schema normalization of various file formats (PDF, spreadsheets) into a standardized JSON representation that preserves logical structures like tables and headers. Once normalized, the semantic transformation step uses an agent-based framework to enrich the data with summaries and keywords, which are then passed to the chunker to be split by tokens or hierarchical titles to ensure semantic continuity. The final step involves the indexer, which generates embeddings and updates the storage backend using incremental indexing strategies that compare document hashes to ensure only modified content is re-processed. This saves time by preventing re-reads of everything when a new file is inserted. 

When it comes to designing a data ingestion pipeline, a decision has to be made between consistency and throughput. High-throughput systems ingest data as fast as possible, but new data is not immediately available for queries, while highly consistent systems ensure data is visible immediately but slow down ingestion to verify new entries. A balance between throughput and consistency can be achieved through Near Real-Time (NRT) indexing. In this model, the system buffers incoming data in a high-speed memory area and periodically flushes these buffers into the permanent index. The use of a buffer however leads to new risks. Too much data input at once could lead to the system refusing all new data or risk buffer overflow and crashing.

# 9. Memory design in RAG systems (10 pts)
Vector memory provides semantic recall by storing past conversation summaries as mathematical points, allowing the system to remember the general topic of a long-term interaction. Structured memory uses SQL to store long term information, such as a user’s account ID, ensuring that these details are retrieved with 100% accuracy rather than with a guess. Episodic logs act as chronological records of user and system interactions. The trade-off is that vector memory is easy to scale but imprecise, while structured memory is precise but requires a rigid design that is hard to change later. Meanwhile episodic logs can suffer context window saturation, where the history becomes so long that the system can no longer process it all at once, leading to a loss of recent records.

# 10. End-to-end system decomposition (10 pts)
A microservices architecture for RAGFlow requires an API Gateway, a Parsing Service, an Embedding Service, a Retrieval Service, and an LLM Reasoning Service. Stateless services, such as the Parsing Service using DeepDoc and the Embedding Service, do not keep a memory of previous interactions, which allows the system to scale horizontally by simply adding more copies of the service as traffic increases. Stateful services require some way to keep the state consistent across instances and include the Vector Database for semantic search and SQL for data storage. Stateful services are more difficult to scale, requiring more complex scaling strategies like sharding, where data is split into smaller, manageable pieces across different servers.

Components should be loosely coupled and independent to create failure isolation boundaries. If one component fails, the failure should be limited in scope as to not bring the entire system down. For example, if the ingestion service crashes after it was presented with a heavy PDF, the chatting function should continue to function with the existing knowledge base. However, this decoupled architecture leads to latency buildup between systems, and does not prevent components and services becoming completely unresponsive until they are recovered.
